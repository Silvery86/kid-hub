Auth & Session Reference

Kid Hub — Auth Flow & Session Management

Kid Hub has two independent auth boundaries: a parent account session (email + password → JWT access + refresh tokens) and a kid unlock session (2-icon visual pattern → short-lived JWT). A separate parent PIN gates re-entry into parent mode without requiring a full re-login.

Three credential types

  • Parent password — bcrypt-hashed account password, verified at /parent/login. Issues access + refresh cookies.
  • Parent PIN — 4-digit bcrypt-hashed PIN, verified at /parent/pin. Re-issues access cookie without full re-login.
  • Kid pattern — 2-symbol (1–6) bcrypt-hashed visual tap sequence, verified at /unlock. Issues kid session cookie.

Three cookie types

  • parent_access — short-lived JWT (15 min), guards /parent/*
  • parent_refresh — long-lived JWT (30 days), rotates access token; hash stored in DB
  • kid_session — 12-hour JWT, guards /, /dashboard, /games/*, etc.

Middleware routing logic

File: middleware.ts — runs on every matched request before the page renders.

Matcher covers: /, /unlock, /dashboard/:path*, /schedule/:path*, /grades/:path*,
                /homework/:path*, /games/:path*, /math/:path*, /english/:path*,
                /parent/:path*

─── Decision tree ───────────────────────────────────────────────────

1. Is path a kid app surface (/, /dashboard, /games, /schedule, etc.)?
   ├─ YES → check kid_session cookie
   │         ├─ valid JWT (typ=kid-session)?  → allow
   │         └─ missing / invalid?           → delete cookie, redirect /unlock
   │
2. Is path /parent/login or /parent/pin?
   ├─ GET → allow (public)
   └─ POST → rate-limit via Upstash sliding window (blocks if > limit)
             → allow on success
   │
3. Is path /parent/* (protected)?
   ├─ valid parent_access JWT?           → allow
   ├─ no access but valid parent_refresh? → mint new access token, set cookie, allow
   └─ neither valid               → delete both cookies, redirect /parent/login
   │
4. Everything else (/_next/static, /api/*, /unlock) → allow (no matcher)

Important: /parent/pin is treated as a public GET path by the middleware. It relies on verifyPinAction server-side to issue new cookies. The PIN page itself checks for a valid refresh token before rendering — unauthenticated users hitting it directly will be redirected server-side to /parent/login.

Parent auth flow

This flow applies to first-time setup and returning parent logins. The UI at /parent/login auto-detects whether an account exists (checkParentSessionAction) and shows either "Thiết lập tài khoản" (first time) or "Đăng nhập" (returning).

Parent PIN gate (/parent/pin)

The PIN is a secondary, faster gate into parent mode — it avoids requiring the full email/password re-login every session while still protecting the parent dashboard from casual access. After successful email/password login with a PIN already configured, the parent_access cookie is cleared and the parent is redirected to /parent/pin.

1

2

The PIN and the account password are independent. Changing one does not affect the other. The PIN hash lives in the parent_pins table (separate row), while the account password hash is on the users row.

Session renewal & token rotation

Access token renewal

Middleware-level rotation

When a request arrives with an expired parent_access cookie but a valid parent_refresh cookie, the middleware mints a new access token inline and sets it on the response — the page renders without a redirect. This keeps the parent logged in for up to 30 days without any UX interruption.

The middleware does not verify the refresh token hash against the database (edge runtime constraint). Full DB-verified rotation happens in ensureParentSession() inside server actions.

Server-action rotation

Full rotation in ensureParentSession

Server actions that require auth call ensureParentSession(), which:

  1. Verifies parent_access JWT — returns immediately if valid.
  2. Falls back to parent_refresh JWT — verifies signature.
  3. Loads refreshTokenHash + refreshTokenExpiresAt from DB.
  4. Compares raw token against stored bcrypt hash.
  5. If valid: issues brand-new access + refresh tokens, updates DB hash.
Sign-out behaviour

Explicit sign-out

signOutParentAction() reads the raw parent_refresh cookie, verifies its JWT to extract userId, then calls userRepo.clearRefreshToken(userId) to null out the stored hash. Both cookies are deleted from the browser. This means the refresh token is immediately revoked server-side — a stolen token cannot be used after sign-out.

Kid unlock flow

The kid session is completely independent from the parent session. A valid kid_session grants access only to kid routes; it cannot satisfy the parent middleware check.

1
App opens → middleware checks kid_sessionIf valid → continue to requested route. If absent/expired → redirect to /unlock.
2
Kid taps 2 icons in sequence at /unlockEach icon maps to a digit 1–6. Pattern is exactly 2 symbols (e.g. "35").
3
Browser calls verifyKidPatternAction(pattern)Server validates format, then compares against bcrypt hash in users.kidPatternHash.
4
Up to 5 attempts allowed; 30-second lockout on 5th failureLockout state stored in users.kidPatternAttempts + users.kidPatternLockedUntil.
5
On success: kid_session cookie set (12 hours); redirect to /dashboardPattern attempt counter is reset.
Pattern setup (parent-only)

Setting the kid unlock pattern

The parent configures the kid pattern via setKidPatternAction(pattern) from the parent dashboard. This action calls ensureParentSession() first — an unauthenticated caller cannot set or change the kid pattern. The pattern hash is stored in users.kidPatternHash.

Login page auto-detection logic

ParentLoginView calls checkParentSessionAction() and checkParentPinAction() on mount to determine the correct initial state: 

hasSession = true  AND hasPin = true   → redirect /parent (already logged in)
hasSession = true  AND hasPin = false  → skip login, show PIN creation flow
hasSession = false AND hasParentAccount = false → isSignup=true  → button: "Tạo tài khoản"
hasSession = false AND hasParentAccount = true  → isSignup=false → button: "Đăng nhập"

The "Tạo tài khoản" button only appears when no parent account exists in the database. After seeding giang8692@gmail.com as the parent account, the button will correctly show "Đăng nhập" on all subsequent visits.

Security notes

P0 blocker

SESSION_SECRET not set in dev

docker-compose.yml line 31 does not set SESSION_SECRET. Without it, the app throws at startup in dev. Set a 32+ char random value before running locally without .env.local.

Noted

No HTTP-layer rate limit on PIN

verifyPinAction has DB-level lockout (5 attempts, 60 s) but no Upstash rate limiter on the POST handler. Concurrent requests can bypass the lockout. Upstash limiter is already wired for /parent/login posts.

Done

All cookies are HttpOnly + SameSite

All three session cookies are HttpOnly, SameSite=lax, path=/, and Secure in production. Refresh token is bcrypt-hashed before storage — a DB leak does not expose raw tokens.

  • Account password is hashed with bcrypt (12 rounds) — raw passwords never stored.
  • Refresh token is hashed before DB storage; raw token only travels in the cookie.
  • Kid pattern is hashed with bcrypt — the parent cannot read back the pattern from the DB.
  • Kid session cookie has no DB backing — revocation happens only by expiry or manual cookie deletion.
  • A valid kid_session cannot call any parent-gated server action — ensureParentSession() checks only parent tokens.
  • Email is lowercased and trimmed before DB storage and lookup to prevent case-sensitivity bypass.
