Security Review

Kid access needs its own auth boundary.

Current behavior leaves the child experience effectively public: the app root redirects straight to /dashboard, and middleware only guards /parent routes. The right fix is not to reuse parent auth for everything, but to add a separate, lightweight kid session that is easy for a child to unlock while keeping strangers out.

Recommendation

Keep parent authentication as account and password at first use, then issue an access token plus long-lived refresh token so the parent usually stays signed in for about one month. Separately, add a dedicated kid_session and route kids through a simple /unlock screen.

Why this fits the product

Parent setup and administration get stronger identity checks, while day-to-day child access stays fast. The two role boundaries stay separate: parent tokens for /parent, kid unlock for child routes.

Confirmed review findings

Critical

Dashboard access is public

app/page.tsx redirects every visitor to /dashboard, and app/(dashboard)/dashboard/page.tsx renders data without any auth check. This means anyone who opens the app on the device lands in the kid area immediately.

Critical

Middleware only protects parent routes

middleware.ts verifies only the parent_session cookie and only runs on /parent and /parent/pin. No matcher covers /dashboard, /games, or the root redirect.

Medium

Parent auth is strong but scoped narrowly

The existing parent flow is sound for its current job: hashed PIN, signed HttpOnly cookie, and middleware verification. The gap is not missing auth infrastructure; it is missing child-facing route protection.

Useful reuse

There is already a clean auth pattern to copy

server/actions/auth.actions.ts and server/services/auth.service.ts already separate service logic, cookie handling, and server action orchestration. A kid auth flow should follow the same layering with its own cookie and validation rules.

Design goals

  1. Parent must complete account and password sign-in before first-time app use.
  2. Parent should not need to log in often: session renewal target is roughly 30 days.
  3. Kid can reach the dashboard in 2 taps or less after opening the app.
  4. Random people cannot immediately access the kid dashboard from a fresh session.
  5. Parent mode stays strictly stronger than kid mode with full account auth.
  6. Implementation stays simple enough to ship without introducing fragile UX.

Proposed auth model

Area Proposal Reason
Parent first login Require account and password on first run at /parent/login. Establishes clear owner identity before configuring kid access.
Parent session model Use short-lived access token plus refresh token rotation. Improves security while avoiding frequent re-login for the parent.
Refresh lifetime Set refresh token max age around 30 days with revocation support. Matches your requirement: parent usually logs in once per month.
Kid credential Parent configures a 2-step visual secret such as icon A then icon D. Easy for a child, harder for outsiders than open access, simpler than typing.
Kid session Create a separate signed HttpOnly kid_session cookie. Prevents accidental reuse of parent privileges on kid routes.
Protected routes Require kid session for /, /dashboard, and /games/*. Closes the current public entry path.
Unlock route Add public /unlock page with large tiles and no keyboard dependency. Matches the tablet-first, child-friendly product direction.
Session lifetime Kid session can stay daily (8 to 12 hours), parent session renews up to 30 days. Balances child convenience, parent convenience, and stronger parent control.
Parent separation Keep parent access token and refresh token fully separate from kid_session. A kid unlock must never grant parent capabilities.

Parent onboarding and month-long sign-in

1. First launch by parent

Parent opens /parent/login and signs in with account and password. If no account exists yet, parent creates it once.

2. Token issuance

Server returns a short-lived parent access token plus a refresh token stored in a secure HttpOnly cookie.

3. Background renewal

Before access token expiry, refresh endpoint rotates tokens. Parent remains signed in for up to about 30 days unless explicit sign-out or token revocation happens.

4. Child default experience

Child entry still lands on /unlock first. Parent login state does not bypass the kid unlock barrier for regular child use.

Recommended user flow

1. App opens

Request hits middleware. If kid_session is valid, continue to /dashboard. If not, redirect to /unlock by default.

2. Kid unlocks

The child taps two large icons in the configured order. Example: sun then bus. Keep the icon set small, visual, and distinct.

3. Server verifies pattern

A server action validates the pattern, enforces attempt limits, sets kid_session, then redirects to /dashboard.

4. Parent mode remains independent

Entering /parent requires valid parent account auth (access token or refresh flow). A kid session must never satisfy parent middleware.

Why visual pattern is the best first version

  • A second PIN creates confusion because the app already has a parent PIN.
  • Biometrics are device-dependent and add setup complexity the app does not need.
  • A tap pattern is simple to implement with the existing keypad-style UI approach.
  • It is not high-security identity proof, but it is a meaningful improvement over zero gate.

Suggested middleware behavior

if pathname starts with /parent and pathname is not /parent/login:
  require valid parent access token
  if expired and refresh token valid, rotate and continue
  else redirect to /parent/login

if pathname is /parent/login:
  allow public access

if pathname is /unlock:
  allow public access

if pathname is / or /dashboard or /games/*:
  require valid kid_session
  otherwise redirect to /unlock

Security notes

  • Use Argon2id or bcrypt for parent password hashing, never store raw passwords.
  • Store refresh token hashes server-side and rotate on every refresh.
  • On refresh token reuse detection, revoke that token family and force re-login.
  • Use a distinct JWT payload type for kid sessions instead of sharing parent token helpers blindly.
  • Keep both cookies HttpOnly, same-site lax, and path scoped to /.
  • Rate-limit kid unlock attempts, even if the visual pattern is intentionally simple.
  • Do not allow kid session state to call parent-only server actions.
  • Rate-limit parent login and refresh endpoints to reduce credential abuse.

Delivery order

  1. Add parent account/password login with refresh-token rotation support.
  2. Protect parent routes with access plus refresh behavior and 30-day renewal policy.
  3. Add kid session primitives, unlock page, and middleware gates for child routes.
  4. Add parent-only setup UI for kid unlock pattern.
  5. Add end-to-end tests for parent login renewal and kid session gating.

This sequence ensures the owner identity is established first, then keeps daily kid access fast through the unlock screen while parent sign-in remains mostly once per month.
