1. Goals — What Will Be Achieved After Implementation

🗂️ Always-Fresh Design Manifest

A machine-readable design/manifest.json is regenerated on demand. It maps every route in the app to its design source file, viewport coverage, and implementation status. Acts as a single source of truth that both scripts and CI can query.

⚡ One-Command Drift Check

Running pnpm design:check in under 30 seconds prints a clear pass / fail report: which routes are missing design coverage, which design files are undocumented, which are deleted-but-referenced, and which viewport variants are absent.

🎨 Token Compliance Report

A scan of every .tsx file flags raw semantic palette classes (e.g. bg-blue-500 used where bg-btn-primary is the correct token) so visual token drift is caught before it accumulates.

3. The Four Checks Defined

Check 1 — Route → Design Coverage

Reads: DESIGN_TO_CODE.html (Route & Design Status) Reads: app/**/page.tsx (filesystem) Phase 1

Cross-references every app/**/page.tsx path against the Route & Design Status table in docs/features/design/DESIGN_TO_CODE.html. Fails if:

  • A page.tsx route has no row in the MD table (new route not documented)
  • A row in the MD table references a design file that doesn't exist on disk
  • A row marked "Designed + Implemented" has a design file that was deleted

Skips: redirect-only routes (/), explicitly "Code-only" rows, and N/A rows.

Check 2 — Design File Inventory

Reads: DESIGN_TO_CODE.html (Design File Inventory) Reads: design/components/ (filesystem) Phase 1

Scans design/components/*.jsx and checks each file is documented in the Design File Inventory table of DESIGN_TO_CODE.html. Fails if:

  • A .jsx file in design/components/ is not listed in §1 (undocumented design file)
  • A file listed in §1 no longer exists on disk (stale reference)

Skips utility files that are excluded by a configured set: shared.jsx, android-frame.jsx, browser-window.jsx, ios-frame.jsx, design-canvas.jsx, tweaks-panel.jsx.

Check 3 — Viewport Export Coverage

Reads: design/components/*.jsx (AST-lite regex parse) Phase 2

Parses each non-utility design file for its exported viewport components. The expected pattern is a window.Object.assign(window, {...}) block at the end. Warns if:

  • A design file is missing *TabletL (the primary reference viewport) — this is an error
  • A design file is missing *PhoneP (minimum mobile coverage) — this is a warning
  • A design file exports fewer than 3 viewport variants total

This prevents implementation starting without a complete design specification.

Check 4 — Semantic Token Compliance

Reads: components/**/*.tsx, app/**/*.tsx Phase 2 Configurable allowlist

Scans all TSX files for raw Tailwind semantic palette classes where a design token exists. The token map comes from DESIGN_TO_CODE.html. Flags:

  • bg-blue-500 / bg-blue-600 used for primary UI actions → should be bg-btn-primary
  • Direct colour classes on subject-coded elements → should use bg-math, bg-english, etc.
  • Hardcoded radius classes for cards/pills → should use rounded-card, rounded-pill

Does NOT flag: game feedback colours, destructive-action reds, structural neutrals (slate/zinc/gray), or files listed in the allowlist config.

6.3 HTML Drift Report — pnpm design:report

Generates design/drift-report.html — a visual dashboard showing route coverage matrix, viewport heat-map, and token violation list. Useful for onboarding or sprint planning reviews. Not required for CI.

7. Implementation Phases

1

Phase 1 — Manifest + Coverage Checks (Checks 1 & 2)

Estimated effort: 3–4 hours · Value: Immediately catches undocumented design files and new routes without design coverage.

  • Write scripts/design/design-check.config.ts — utility file exclusion list, route skip list (code-only, N/A)
  • Write scripts/design/build-manifest.ts — parse docs/features/design/DESIGN_TO_CODE.html tables, scan design/components/, scan app/**/page.tsx, output design/manifest.json
  • Write scripts/design/check-coverage.ts — read manifest, run Checks 1 and 2, print results, exit 0/1
  • Add "design:manifest" and "design:check" to package.json
  • Add design/manifest.json and design/drift-report.html to .gitignore
  • Run pnpm design:check — fix the parent-v2.jsx documentation gap that it finds
2

Phase 2 — Viewport + Token Checks (Checks 3 & 4) + HTML Report

Estimated effort: 3–4 hours · Value: Catches design gaps before implementation starts; surfaces token drift.

  • Write scripts/design/check-viewports.ts — regex-parse each non-utility design .jsx for exported viewport function names matching *PhoneP, *TabletL, *Desktop
  • Write scripts/design/check-tokens.ts — scan .tsx files for flagged palette classes against the allowlist in the config
  • Write scripts/design/report.ts — produce a self-contained HTML file summarising all check results with colour-coded status
  • Update design:check script to run all four checks
  • Add "design:report" to package.json
3

Phase 3 — GitHub Actions CI Integration

Estimated effort: 1 hour · Value: Enforces checks on every PR — no more manual checklist.

  • Create .github/workflows/design-check.yml — runs pnpm design:check on push to main and on pull_request
  • Upload drift-report.html as a workflow artifact (downloadable from the Actions tab for visual review)
  • Optionally post a PR comment summarising violations (using actions/github-script)

8. Token Allowlist — What Is and Isn't a Violation

The token check needs to be precise — too strict and it becomes noisy; too loose and it's useless. The config file defines the rules.

Tokens that should be flagged (violation)

Raw class detectedShould use insteadApplies to
bg-blue-500, bg-blue-600 (on CTA buttons)bg-btn-primaryInteractive button backgrounds
text-blue-600 (as primary brand text)text-btn-primaryBrand accent text
rounded-2xl / rounded-3xl (on content cards)rounded-cardSemantic card containers
rounded-full (on pill tags)rounded-pillStatus and label pills

Tokens that are ALLOWED (not a violation)

PatternReason allowed
bg-red-*, text-red-*, border-red-*Error states, destructive actions, game wrong-answer feedback — no semantic token exists for these
bg-green-*, text-green-*Game correct-answer feedback, success states — no semantic token
bg-yellow-*, bg-amber-*Game celebration, warning states — intentional
bg-slate-*, bg-zinc-*, bg-gray-*, bg-stone-*Structural neutral colours — acceptable since they're not semantic
opacity-* + any colourTinting / layering utilities — not semantic violations
Any class in files matching e2e/**, design/**, scripts/**Non-production code — excluded from scan
Files listed in design-check.config.ts tokenAllowlistSpecific files explicitly permitted (e.g., existing Badge.tsx pending design decision)
 
// scripts/design/design-check.config.ts
export const config = {
  /** .jsx files in design/components/ excluded from Check 2 */
  utilityDesignFiles: [
    'shared.jsx', 'android-frame.jsx', 'browser-window.jsx',
    'ios-frame.jsx', 'design-canvas.jsx', 'tweaks-panel.jsx',
  ],

  /** Routes excluded from Check 1 (redirect-only, code-only, etc.) */
  skipRoutes: ['/', '/kid-unlock'],

  /** Semantic-colour classes to flag in Check 4 */
  semanticViolations: [
    { pattern: /bg-blue-(5|6)00/,   suggestion: 'bg-btn-primary'  },
    { pattern: /text-blue-(5|6)00/, suggestion: 'text-btn-primary' },
    { pattern: /rounded-(2|3)xl/,   suggestion: 'rounded-card'     },
  ],

  /** Files exempt from Check 4 (existing / pending design decision) */
  tokenAllowlist: [
    'components/ui/Badge.tsx',
    'components/grades/GradeCard.tsx',
    'components/grades/GradeTierBadge.tsx',
  ],
}
The allowlist approach prevents false positives on day one. Existing known violations are allowlisted so the initial run passes. As the codebase is cleaned up, entries are removed from the allowlist — tightening the check progressively without blocking work.
