1. Design Tokens — The Only Colours Allowed

2. Border Radius Tokens

Raw classes to avoid on semantic elements: Use rounded-card instead of rounded-2xl or rounded-3xl for card containers. Use rounded-pill instead of rounded-full for pill-shaped badges. Raw rounded-full is still acceptable for truly circular elements (icon buttons, avatars).

3. Tap Target Sizes

Kid Hub targets 5–8 year-olds on tablets. All interactive elements must meet these minimum sizes.

TokenTailwind classSizeUse for
--spacing-tapmin-h-tap / min-w-tap3rem (48px)Standard buttons, list items, nav icons
--spacing-tap-lgmin-h-tap-lg / w-tap-lg4rem (64px)Primary action buttons, game answer tiles

Never use h-8 (32px) or smaller for any interactive element. Use min-h-tap as the floor.

6. Usage Rules — Do & Don't

✅ DO — Use semantic tokens for primary actions

bg-btn-primary hover:bg-btn-primary-hover

✅ DO — Use rounded-card for content panels

<div className="rounded-card bg-white p-6">

✅ DO — Use subject tokens for subject UI

bg-math text-white  (not bg-blue-500)

✅ DO — Use text-text-* for body text

text-text-primary text-text-secondary

✅ DO — Use min-h-tap for tap targets

<button className="min-h-tap px-6">

❌ DON'T — Use raw palette for semantic blue

bg-blue-500  ←  violates Check 4

❌ DON'T — Add tokens to :root

:root { --color-x: red }  ←  use @theme {}

❌ DON'T — Hard-code semantic colors in games

bg-green-400  ←  OK for feedback, NOT for subject UI

❌ DON'T — Use tiny tap targets

h-8 (32px)  ←  too small for a 5-year-old

❌ DON'T — Import from server/ in components

import { db } from '@/server/...'  ←  crashes client bundle

Exceptions that ARE allowed (by design)

PatternReason
bg-red-*, text-red-*Error states, destructive actions (sign-out), game wrong-answer feedback
bg-green-*, text-green-*Game correct-answer feedback, inline success
bg-yellow-*, bg-amber-*Game celebration, star/badge highlights, warning states
bg-slate-*, bg-zinc-*, bg-gray-*Structural neutral surfaces — not semantic
rounded-full on circlesCircular avatar images, icon buttons — not pill shapes

7. Viewport System

Kid Hub is tablet-first. The primary reference viewport is Tablet Landscape (1280×800). Phone portrait is the minimum mobile baseline.

ViewportDimensionsDesign suffixTailwind breakpoint
Phone Portrait390×844*PhonePdefault (no prefix)
Phone Landscape844×390*PhoneLlandscape: + max-sm:
Tablet Portrait820×1180*TabletPmd: (≥768px)
Tablet Landscape1280×800*TabletL ⭐lg: (≥1024px)
Desktop1440×900*Desktopxl: (≥1280px)
Always start from *TabletL when implementing a new route. This is the golden reference. Phone variants simplify from it, desktop variants expand.

Orientation variants (RESP-001)

Use portrait: and landscape: Tailwind prefixes defined in globals.css:

<!-- Show only in landscape -->
<div className="portrait:hidden landscape:block">

<!-- Or use the fallback CSS classes -->
<div className="landscape-only">    {/* hides in portrait */}
<div className="portrait-only">    {/* hides in landscape */}

9. Design Automation Checks

Run pnpm design:check to verify design system compliance. All four checks must pass before merging a PR.

Available commands

CommandWhat it doesWhen to run
pnpm design:check Runs all 4 checks (manifest + coverage + viewports + tokens). Exits 1 on any failure. Before every PR merge — also runs automatically in CI
pnpm design:manifest Regenerates design/manifest.json from DESIGN_TO_CODE.md and filesystem. After adding a new route or design file
pnpm design:report Generates design/drift-report.html — a visual dashboard of all check results. Sprint reviews, onboarding, before major refactors

The four checks

#CheckPasses whenFix
1 Route → Design Coverage Every app/**/page.tsx has a design file entry in DESIGN_TO_CODE.md §5 Add a row to §5 or mark as Code-only
2 Design File Inventory Every design/components/*.jsx is listed in DESIGN_TO_CODE.md §1 Add a §1 row or add to utilityDesignFiles in config
3 Viewport Export Coverage Every non-utility design file exports a *TabletL variant Add the missing viewport to the design file, or add to viewportExceptions
4 Semantic Token Compliance No TSX file uses bg-blue-500/600 or text-blue-500/600 for semantic purposes Replace with bg-btn-primary / text-btn-primary, or add to tokenAllowlist

Updating the config

All check behaviour is controlled by scripts/design/design-check.config.ts:

// To exclude a new utility design file from Check 2:
utilityDesignFiles: ['my-utility.jsx', ...],

// To skip a route from Check 1:
skipRoutes: ['/', '/kid-unlock', '/my-code-only-route'],

// To allowlist a file with an existing token violation:
tokenAllowlist: ['components/ui/MyLegacyComponent.tsx', ...],

// To exempt a single-viewport design file from Check 3:
viewportExceptions: ['dashboard-v2.jsx', ...]

10. AI Prompt Library

Short, paste-ready prompts for common design system tasks. Each prompt is self-contained — no extra context needed.

🔍 Run full design check

Use: verify compliance before merging
Run `pnpm design:check` and report the result. If any check fails, list each failure with its fix instruction from the output. Do not edit any files — only report.

🗺️ Add new route to design tracking

Use: after creating a new app/[route]/page.tsx
I added a new route at [ROUTE] (file: [PAGE_PATH]). Update DESIGN_TO_CODE.md §5 by adding a new row for this route. The design source is [DESIGN_FILE or "—" if code-only] and the status is [Designed + Implemented / Code-only / Design-only]. Then run `pnpm design:check` to verify Check 1 passes.

📁 Register new design file

Use: after adding a new .jsx file to design/components/
I added a new design file `design/components/[FILENAME].jsx`. Add a row to DESIGN_TO_CODE.md §1 with: the filename, the routes it covers, and its viewport variants. Then run `pnpm design:check` to verify Checks 2 and 3 pass.

🎨 Fix semantic token violations

Use: when Check 4 reports violations
Run `pnpm design:check`. For each Check 4 violation reported, replace the raw palette class with the suggested token class. Rules: `bg-blue-500` / `bg-blue-600` → `bg-btn-primary`. `text-blue-500` / `text-blue-600` → `text-btn-primary`. Do NOT change game feedback colours (red/green/yellow) or structural neutrals (slate/zinc/gray). After fixing, run `pnpm design:check` again to confirm 0 violations.

📊 Generate drift report

Use: sprint review or onboarding
Run `pnpm design:report` and confirm the output says "Report written → design/drift-report.html". Then open the file in a browser and summarise: (1) how many routes have design coverage, (2) how many design files are documented, (3) any token violations found.

🚫 Allowlist a known violation

Use: when a violation is intentional and cannot be fixed now
Add `[FILE_PATH]` to the `tokenAllowlist` array in `scripts/design/design-check.config.ts`. Add a comment explaining why it's allowlisted. Then run `pnpm design:check` to confirm it passes.

🔧 Fix Check 3 — missing viewport

Use: when a design file is missing a required viewport export
Check 3 reports that `design/components/[FILE].jsx` is missing a *TabletL viewport. Either: (a) add the missing viewport component and export it in the Object.assign(window, {...}) block, or (b) if this file intentionally has fewer viewports, add '[FILE].jsx' to the `viewportExceptions` array in `scripts/design/design-check.config.ts`. Run `pnpm design:check` to confirm.

✅ Verify design system before PR

Use: final check before opening a pull request
Before I open this PR, run the following and report results for each: 1) `pnpm design:check` — all 4 checks must pass. 2) `pnpm format:check` — no formatting errors. 3) `pnpm lint` — no lint errors. If any fail, list what needs to be fixed. Do not auto-fix unless I ask.
